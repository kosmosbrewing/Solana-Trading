## 5. effectiveRR 구조 분석

### 현재 공식
```
effectiveRR = max(rewardPct − roundTripCost, 0) / (riskPct + roundTripCost)

rewardPct     = (TP2 − entry) / entry        ← ATR(20) × 3.5
riskPct       = (entry − stopLoss) / entry   ← entry − candle.low
roundTripCost = entryImpact + exitImpact + ammFee(0.5%) + mevMargin(0.15%) ≈ 0.65%
```

### 기준값
| effectiveRR | 결과 |
|-------------|------|
| < 1.2 | **Reject** (체결 거부) |
| 1.2 ~ 1.5 | Accept (50% 사이즈 감량) |
| ≥ 1.5 | Full pass (100% 사이즈) |

### 핵심: TP2는 ATR(20) × 3.5이고, TP1(ATR × 1.5)은 effectiveRR 계산에 사용되지 않음

- effectiveRR은 TP2 기준으로 계산되므로, TP2까지의 보상이 SL + 비용 대비 충분해야 함
- 현재 토큰에서 ATR(20) × 3.5가 candle.low SL 대비 충분한 보상을 주지 못하는 구조

## 6. 진단 — effectiveRR이 구조적으로 낮은 원인 후보

1. **roundTripCost 0.65%가 과대추정** — 실제 slippage가 0.65%보다 작다면 effectiveRR이 즉시 올라감
2. **TP2 = ATR(20) × 3.5 vs SL = candle.low 비율 불균형** — ATR 기반 reward가 SL 거리 대비 구조적으로 작을 가능성
3. **대상 토큰의 변동성 부족** — 저가 meme 토큰의 ATR이 좁아 reward가 작음
4. **표본(토큰 universe) 단일 집중** — 시그널 발생 토큰이 1개이므로 판단 자체가 편향

### 추가 검토 메모 — 현재 코드 기준에서 먼저 확인할 점

후속 코드 리뷰 기준으로, 위 1~4는 여전히 유효한 가설이지만 **패치 우선순위는 보정이 필요**하다.

#### A. pre-gate execution viability가 실제 주문 크기 이전에 차단을 끝내고 있다

- 현재 `processSignal()`은 `gateResult.rejected`면 즉시 반환한다.
- 따라서 live 경로에서 `poor_execution_viability`가 gate 단계에서 걸리면,
  실제 리스크 사이징 이후의 `size-aware execution check`까지 도달하지 못한다.
- 즉 이번 카나리아의 직접 blocker는 "post-size actualExecution"이 아니라
  **gate 단계의 pre-size execution viability**다.

코드 근거:
- `src/orchestration/signalProcessor.ts` — `gateResult.rejected` early return
- `src/orchestration/signalProcessor.ts` — post-size `evaluateExecutionViabilityForOrder()`는 그 이후에만 실행

#### B. pre-gate probe 수량 단위가 실제 주문 단위와 다를 가능성이 있다

- `evaluateExecutionViability()`는 인자로 `estimatedQuantitySol`을 받지만,
- 이를 그대로 `buildVolumeSpikeOrder(..., quantity)`에 넣는다.
- 하지만 `Order.quantity`는 token units 기준이다.
- 반면 early probe용 `estimatedPositionSol`은 `realtimeHandler.ts`에서 SOL notionals처럼 계산된다.

해석:
- 현재 pre-gate RR은 **실제 체결 주문과 다른 크기/단위의 probe**로 계산되고 있을 가능성이 있다.
- 따라서 `roundTripCost 0.65%`나 `TP2/SL` 비율만 먼저 조정하면,
  진짜 blocker가 비용 모델인지, pre-gate probe 계산 자체인지 분리하지 못할 수 있다.

코드 근거:
- `src/orchestration/realtimeHandler.ts` — `estimatedPositionSol`
- `src/gate/executionViability.ts` — `evaluateExecutionViability()`
- `src/utils/types.ts` — `Order.quantity // token units, not SOL notional`

#### C. edge tracker blacklist는 부차 이슈가 아니라 cadence 회복 경로의 일부다

- 3건은 `poor_execution_viability`가 아니라 `Pair blacklisted by edge tracker`로 차단됐다.
- 따라서 RR 문제만 완화해도 동일 pair가 blacklist 상태면 cadence는 즉시 회복되지 않을 수 있다.
- 현재 scanner cooldown은 evicted token 재진입만 다루며,
  blacklisted pair의 재유입 억제와는 별개다.

코드 근거:
- `src/risk/riskManager.ts` — pair blacklist reject
- `src/reporting/edgeTracker.ts` — blacklist 조건
- `src/scanner/scannerEngine.ts` — re-entry cooldown은 evicted token 기준
