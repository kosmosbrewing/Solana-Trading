## Priority 1

`pool discovery` 포화를 먼저 줄인다.

- 최근 7시간에 `Pool discovery queue overflow`가 419회 발생했다.
- 이 경고는 throttle이 걸려 있으므로, 사실상 거의 상시 포화로 해석된다.
- discovery 단계가 막히면 이후 gate 품질 이전에 표본 자체가 잘리게 된다.

## Priority 2

watchlist 재순환과 candidate churn을 줄인다.

- 소수 토큰이 반복 재유입되며 watchlist `add / evict` churn을 만들었다.
- 표본이 넓게 퍼지지 않고 동일 토큰 재순환에 많이 소모됐다.
- 이는 execution 문제보다 admission / reentry cadence 문제에 가깝다.

## Priority 3

RR / gate 완화보다 먼저 reject 원인을 더 잘 분리해 측정한다.

- 이번 7시간에서 `poor_execution_viability`는 1건뿐이었다.
- 현재 runtime 기준에서는 이 reject가 정책과 일치한다.
- 따라서 지금 threshold를 감으로 풀기보다, execution viability가 진짜 blocker인지 discovery / admission이 더 큰 blocker인지 telemetry로 분리해야 한다.

## Missing Gates

현재 시점에서 사명 달성 경로로 넘어갔다고 보기 어려운 이유는 단순히 trade 수가 적어서만이 아니다.
아래 게이트를 아직 판정할 수 없거나, 판정 근거가 충분하지 않다.

- `Mission Gate`: 최근 `50 executed trades` 기준 explained entry ratio, source attribution completeness, context -> trigger consistency를 확인할 표본이 없다.
- `Execution Gate`: 최근 `20 measured trades` / `20 measured exits` 기준 quote decay, sell impact realism을 읽을 표본이 없다.
- `Edge Gate`: live 구간에서 expectancy / net PnL / PF를 해석할 수 있는 trade sample이 없다.

즉 현재 문제는 "수익이 아직 안 났다"보다, Mission / Execution / Edge를 같은 프레임으로 읽을 `Gate-Proven Sample`이 아직 없다는 점이다.

## Why Paper Still Matters

사명 달성 관점에서 live cadence 복구만으로는 부족하다.
현재 active plan 기준으로는 paper validation이 여전히 선행 조건이다.

- paper에서 `expectancy after fees/slippage`를 다시 확인해야 한다.
- paper에서 `quote decay`, `gate rejection mix`, `hold time / exit reason distribution`, `explained entry ratio`를 운영 가능한 방식으로 쌓아야 한다.
- paper sample이 `50 trades` 미만이면 live gate를 통과했다고 볼 수 없다.

따라서 이 메모의 우선순위 3개는 live canary 해석용 운영 우선순위이지,
paper validation을 대체하는 항목은 아니다.

## Next Review Criteria

다음 canary 판정은 아래 조건이 모였을 때 다시 내리는 것이 맞다.

1. discovery overflow와 watchlist churn이 줄어들어 signal cadence가 의미 있게 회복된다.
2. `poor_execution_viability`가 unique token 기준으로 반복 blocker인지, 아니면 소수 사례인지 분리된다.
3. paper에서 live enablement 판단에 필요한 최소 표본과 운영 품질 지표가 다시 확보된다.
4. 운영 개입 없이 일정 시간 유지 가능하다는 근거가 쌓인다.
5. `internal_activity` candidate path가 실제 runtime diagnostics / watchlist에 관측된다.

## Working Conclusion

지금 우선순위는 아래 순서가 맞다.

1. discovery throughput 복구
2. watchlist churn / reentry cadence 안정화
3. reject telemetry 분리 후 execution viability 재판단

즉 지금은 전략 튜닝보다 운영 cadence를 먼저 살려 `Gate-Proven Sample`까지 올라가는 것이 맞다.
