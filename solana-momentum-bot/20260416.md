## Mission Quality Verdict

### A. Source-of-truth integrity: RED

현재 가장 심각한 문제.

- 실제 운영 체결이 있었는데 로컬 trade snapshot이 그걸 못 본다
- synced signal artifact와 사용자 체결 사실이 충돌한다
- 이 상태에서는 expectancy, win rate, 50 canary trades 진행률을 믿을 수 없다

사명 기준 의미:

- edge가 있는지 없는지보다 먼저
- `무슨 거래가 실제로 일어났는지`조차 재현되지 않는다

### B. Flow quality: RED-AMBER

- signal은 `91`
- 하지만 persisted artifact 기준 체결은 `0`
- 사용자 제보 기준 실제 체결은 있었음

즉 flow는 `완전 부재`가 아니라 `관측/원장 불일치` 상태다.
이건 low-frequency 문제보다 더 위험하다.

### C. Breadth / freshness: AMBER

- idle eviction은 동작
- candidate turnover도 있음
- 하지만 `idleSkip`가 아직 매우 크고
- unique signaled tickers가 `6`에 불과

즉 문제 1은 완전히 닫히지 않았다.

### D. Edge quality: UNKNOWN

현재는 `edge가 좋다/나쁘다`를 단정할 수 없다.

이유:

- 실제 profitable trade가 있었다는 사용자 사실이 있음
- 그러나 로컬 trade dump는 stale
- signal artifact는 gate_rejected

즉 edge 품질보다 먼저 `integrity recovery`가 선행되어야 한다.

## Direct Answer: “매수가 한 번밖에 없는데 신호가 너무 적은가?”

부분적으로만 맞다.

- `signal` 자체는 적지 않았다 (`91`)
- 하지만 **실제 buy 표본은 지나치게 적거나 추적 불가능하다**

그래서 정확한 표현은 이렇다.

- `signal scarcity`보다
- **`execution conversion + ledger integrity scarcity`가 문제**

## Improvement Plan

## P0. Integrity Recovery

이게 최우선이다. 전략 튜닝보다 먼저.

### P0-1. `sync-vps-data.sh` 하드닝

문제:
- 현재 `VPS_DATABASE_URL` 또는 `pm2 env 0`에 의존
- 잘못된 DB를 덤프해도 로컬은 정상처럼 보임

개선:
- `pm2 env 0` 대신 `momentum-bot` 이름 기준 DB URL 확인
- dump 직후 아래를 강제 출력
  - DB host / dbname
  - `max(created_at)`
  - `max(closed_at)`
  - row count
- latest session time보다 DB 최신 시각이 지나치게 오래되면 실패 처리

완료 기준:
- 로컬 `vps-trades-latest.jsonl`가 실제 운영 DB 최신 시각과 맞는다

### P0-2. tx-success append-only fallback ledger 추가

문제:
- 현재 live buy는 `executeBuy()` 후 `insertTrade()` 순서
- tx는 성공했는데 DB insert가 실패하면 체결 사실이 사라질 수 있음

개선:
- `executeBuy()` 성공 직후 append-only JSONL 기록
- 필드:
  - `txSignature`
  - `strategy`
  - `pairAddress`
  - `tokenSymbol`
  - `signalId`
  - `positionId`
  - `createdAt`
  - `plannedEntryPrice`
  - `actualEntryPrice`

완료 기준:
- DB가 실패해도 executed buy 사실은 파일 기준으로 복구 가능

### P0-3. tx success -> DB row invariant 추가

문제:
- live tx가 있는데 DB row가 없을 수 있는 구조

개선:
- `insertTrade()` 실패 시
  - critical notifier
  - dedicated error log
  - trading halt
  - reconciliation required marker

완료 기준:
- `on-chain buy success but ledger missing`이 조용히 지나가지 않음

### P0-4. signal/execution/reconciliation key 정렬

지금 필요한 연결키:

- `signalId`
- `positionId`
- `tradeId`
- `txSignature`

개선:
- signal intent
- notifier open/close
- DB row
- fallback executed-trades ledger

이 네 경로에서 위 키를 전부 남긴다.

완료 기준:
- tx 하나를 넣으면 signal → open → close → DB row를 바로 역추적 가능
