## 4. pure_ws_breakout — 설계는 convex, 실측은 flip-cutter

### 4.1 런타임 파라미터 (`tradingParams.ts:329-340`)

| 단계 | 파라미터 | 값 |
|---|---|---|
| PROBE | `pureWsProbeWindowSec` | **30s** |
| PROBE | `pureWsProbeHardCutPct` | **-3%** (MAE) |
| PROBE | `pureWsProbeFlatBandPct` | ±10% |
| PROBE | `pureWsProbeTrailingPct` | 3% |
| T1 | `pureWsT1MfeThreshold` | **+100% (2x)** |
| T1 | `pureWsT1TrailingPct` | 7% |
| T2 | `pureWsT2MfeThreshold` | **+400% (5x)** |
| T2 | `pureWsT2TrailingPct` | 15% |
| T2 | `pureWsT2BreakevenLockMultiplier` | **entry × 3** (3x lock) |
| T3 | `pureWsT3MfeThreshold` | **+900% (10x)** |
| T3 | `pureWsT3TrailingPct` | 25% |
| T3 | time stop | **없음 (no time stop)** |

### 4.2 설계 의도 — 사명 부합

- `T3 no-time-stop` + `T2 entry×3 lock` + `T3 trail 25%` 조합은 **명백히 convex tail 설계**.
- "Phase 1 burst 로 진입 → T1 에서 일부 살림 → T3 로 runner" 는 mission-refinement §5 의 "5x+/10x+ winner 분포" 와 직접 부합.

### 4.3 실측 (04-16 → 04-22, 7일)

```
closed=83  net=-0.0231 SOL  logGrowth=-0.0234
maxDD=3.06%  winners 5x+ (net)=0  winners 10x+ (net)=0
```

- 83 trade 중 **net 5x+ = 0**. T2 visit 이 있었는지조차 현재 ledger 에서 확인 불가 (MFE peak 필드 별도 저장 필요).
- Exit 분포 (executed-sells 전체 127 중 pure_ws 비중): `REJECT_TIMEOUT` 82% / `WINNER_TIME_STOP` + `WINNER_TRAILING` 소수.
- 즉 **"ticket 을 T1 까지 살려보내는 전 단계에서 대다수 cut"**.

### 4.4 조기 cut 을 만드는 gate chain

| 요인 | 값 | 효과 |
|---|---|---|
| PROBE window | 30s | Phase 2 consolidation (30s~3min) 를 flat 으로 오판 |
| PROBE hardcut | MAE ≤ -3% | 초기 dip 에서 hard cut |
| PROBE flat band | ±10%, 30s 만료 시 close | consolidation → REJECT_TIMEOUT |
| quickRejectClassifier | 45s, 2+ factor degraded → exit | MFE decay + buy ratio drop + tx density drop 조합 exit |
| holdPhaseSentinel | 3 factor (buy pressure collapse / tx density drop / peak drift) 2+ → `DEGRADED_EXIT` | T1/T2/T3 보유 중에도 microstructure 악화 시 조기 exit |

→ **설계의도의 "T3 no-time-stop" 에 도달하려면 위 5개 gate 를 전부 통과해야 하는데, 30s PROBE + hardcut 조합에서 대부분 전사**.

### 4.5 구조적 진단

- 설계는 convex. 파라미터는 flip-cutter.
- 이는 **사명 불일치가 아니라 파라미터 mis-calibration + Stage 1 Safety Pass 원칙의 부산물**.
- `P4. Hard Safety Never Compromised` 가 convex 보다 위에 있기 때문에, bleed 억제 (30s PROBE, -3% hardcut) 가 convex 포획을 희생한다.

## 5. "1분 안에 fire 해야 먹는 구조" 의 구조적 근거

논의 맥락 (본 리포지토리 외부 피드백에서 제기된 질문) 과 현재 코드를 맞춰보면:

### 5.1 "signal fire" 의 의미

- "fire" = **signal fire** = trigger 발화 시점 (`bootstrap_10s` VolumeMcapSpikeTrigger 또는 V2 `evaluateWsBurst`).
- 독립 전략명이 아님. 코드 전체에서 `fire` 는 `fire-and-forget` (async 패턴), `signal fire / trigger fired` (발화 시점), `over-fire` (false positive rate) 세 용법만 존재.
- **즉 "fire 전략" ≠ 위 3개 lane 과 무관한 별도 전략이 아니라, pure_ws/cupsey lane 의 상태기계 구조를 설명하는 용어**.

### 5.2 두 층의 latency

**층 1 — Detection Latency**

```
Pool swap burst → Helius WS (~T+0~1s) → V1/V2 Scanner 집계 (~T+3s)
→ Signal Fire (~T+5-10s) → entryDriftGuard (Jupiter quote)
```

- 늦게 fire 되면 `entryDriftGuard (maxFavorableDriftPct=0.20)` 가 reject — **구조적으로 "쫓아가기" 차단**.

**층 2 — Probe Window**

```
Entry → 30s PROBE (pure_ws) / 45s PROBE (cupsey) → MFE 검증
```

- pure_ws: MFE < +100% 면 trail/flat/hardcut 중 하나로 cut.
- cupsey: MFE ≥ +2% → WINNER 승격, 이후 12분 안에 4% trailing 으로 close.

### 5.3 Phase 3 breakout blind spot

Memecoin burst 의 전형적 4 phase 대비 엔진 반응:

| Phase | 시간 | 가격 | cupsey 반응 | pure_ws 반응 |
|---|---|---|---|---|
| 1. Initial pump | 0-30s | +3~+20% | PROBE pass → WINNER | PROBE pass → T1 후보 |
| 2. Consolidation | 30s-3min | -5%~+5% flat | PROBE timeout 또는 WINNER 4% trail | **PROBE 30s flat → REJECT_TIMEOUT (치명)** |
| 3. Breakout | 3-30min | +50%~+500% | 이미 12min time stop 으로 close 됐을 수 있음 | **포지션 없음 (구조적 miss)** |
| 4. Trend/fade | 30min+ | ±% | 새 signal fire 시 재진입 (300s v1 per-pair cooldown) | 새 signal fire 시 재진입 (1800s cooldown, 04-22 확장) |

→ **두 lane 다 Phase 3 breakout 을 구조적으로 miss 할 가능성이 높고, "Phase 3 재등장 burst 로 간접 재포획" 은 희망적 시나리오이지 관측된 사실이 아님**.

## 6. Missed Alpha Observer — 이미 구현 완료

외부 피드백에서 "Stage 2 즉시 구현" 으로 제안된 Missed Alpha Observer 는 **이미 선행 구현됨**.

- 파일: `src/observability/missedAlphaObserver.ts` (416 줄, 2026-04-22)
- 기록 파일: `${REALTIME_DATA_DIR}/missed-alpha.jsonl`
- 동작: reject 이벤트 (survival / entry_drift / sell_quote / viability / security / v2_pass) 발생 시 **T+60s / T+300s / T+1800s** Jupiter quote 로 signal price 대비 delta 기록
- 설계 원칙: fire-and-forget / 독립 429 circuit / per-tokenMint 30s dedup / inflight cap 50
- 문서화된 원칙: *"NOT a trading decision input — pure observability. threshold 튜닝은 최소 1 week 분포가 쌓인 후 사람이 판단 (Stage 1 관측 우선)"*

### 6.1 현재 Observer 의 coverage gap

Observer 는 **entry 전에 reject 된 signal** 의 post-reject trajectory 만 추적한다. 반면 상대 논의의 핵심 시나리오 ("진입 → 30s flat cut → Phase 3 breakout miss") 는 **이미 진입한 trade 의 post-exit MFE** 추적이 필요하다. 현재 구현에는 아래 카테고리가 빠져 있다:

- `probe_reject_timeout` (PROBE 30s 만료 후 flat cut)
- `probe_hard_cut` (MAE < -3%)
- `probe_flat_cut` (±10% band 이탈)
- `quick_reject_classifier_exit`
- `hold_phase_sentinel_degraded_exit`

→ Phase 3 winner miss 비율을 정량 평가하려면 이 5종 확장이 필요. 예상 작업량 1-2시간.
