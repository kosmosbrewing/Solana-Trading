## 4. What Has Already Been Fixed

2026-04-16 기준 cupsey lane에는 아래 보강이 이미 들어갔다.

### 4.1 shared ledger, separate control

- cupsey buy 직후 `OPEN` trade row를 공용 `trades`에 즉시 적재
- 대신 cupsey OPEN row는 공용 monitor / tick exit / main risk portfolio에서 제외

의미:

- 원장은 공유
- 제어권은 cupsey state machine이 소유

### 4.2 sell failure no longer fakes closure

기존 문제:

- live sell 실패 시에도 `CLOSED` 처리될 수 있었음

수정 후:

- sell 실패 시 `OPEN` 유지
- `critical` notifier 발송
- close notifier 미발송

### 4.3 close persistence failure no longer emits fake close

기존 문제:

- sell은 성공했는데 DB close 실패 시에도 종료처럼 보일 수 있었음

수정 후:

- `cupsey_close_persist` critical 발송
- DB close 성공 전에는 close notifier 미발송

### 4.4 cupsey restart recovery added

기존 문제:

- 재기동 후 in-memory cupsey state를 잃음

수정 후:

- startup 시 공용 `trades`의 `OPEN cupsey_flip_10s`를 다시 읽어 cupsey state machine으로 rehydrate
- `highWaterMark`, `trailingStop`를 DB에 계속 저장하여 recovery 품질 개선

주의:

- 과거에 열린 오래된 cupsey OPEN row는 peak/trailing 정보가 부족할 수 있어 보수적으로 복구될 수 있다
- 완전한 state replay는 아직 아님

## 5. Direct Answer To The Core Question

### “매수가 한 번밖에 없는데 신호가 너무 적은가?”

정확히는 아니다.

- `signal`은 적지 않았다
- **실제 buy conversion과 원장 일치성이 부족했다**

즉 현재 핵심 문제는:

- `signal scarcity`
가 아니라
- `execution conversion scarcity + ledger integrity scarcity`

### “signal 나오면 전부 매수하면 더 유리한가?”

아니다.

이유:

- `bootstrap_10s`는 메인 execution lane이 아님
- bootstrap은 signal source이고, cupsey가 그 위에서 다시 quality / timing을 걸러야 한다
- `all-buy bootstrap`은 gate / timing layer를 제거해 저품질 체결까지 다 사게 만들 위험이 크다

## 6. Execution Plan

## P0. Integrity Hardening

전략 튜닝보다 먼저, source-of-truth를 고정한다.

### P0-1. `sync-vps-data.sh` hardening

목표:

- 실제 `momentum-bot`가 쓰는 운영 DB를 정확히 덤프
- stale dump를 성공처럼 보이지 않게 함

필수 출력:

- DB host / dbname
- `max(created_at)`
- `max(closed_at)`
- row count
- `momentum-bot` pm2 app name 기준 DATABASE_URL auto-resolve

실패 조건:

- latest session timestamp 대비 DB 최신 시각이 과도하게 오래되면 sync 실패 처리
- preflight `max(created_at)`와 dumped JSONL `dump_max_created`가 다르면 sync 실패 처리
- override가 꼭 필요할 때만 `ALLOW_STALE_DB_DUMP=true`

### P0-2. append-only executed ledger

목표:

- DB insert 실패와 무관하게 실제 buy/sell tx 사실을 보존

필드 최소셋:

- `txSignature`
- `strategy`
- `pairAddress`
- `tokenSymbol`
- `signalId`
- `positionId`
- `tradeId` (가능한 경우)
- `plannedEntryPrice`
- `actualEntryPrice`
- `recordedAt`

### P0-3. tx success -> DB row invariant

목표:

- on-chain tx success 이후 ledger row 부재를 조용히 넘기지 않음

필수 동작:

- critical notifier
- dedicated error log
- reconciliation required marker
- 필요 시 live halt

### P0-4. trace key alignment

다음 키를 전체 경로에서 정렬한다.

- `signalId`
- `positionId`
- `tradeId`
- `txSignature`

경로:

- signal intent
- executed ledger
- notifier open / close
- DB trades row

완료 기준:

- tx 하나로 signal -> open -> close -> DB row를 역추적 가능

## P1. Execution Funnel Visibility

24h 기준 아래 funnel을 고정 출력한다.

1. `signals`
2. `bootstrap gate pass`
3. `cupsey gate pass`
4. `stalk entered`

## P2. Gate Review

integrity 복구 후에만 수행한다.

우선 재검토 대상:

- `security gate`
- `poor_execution_viability`
- cupsey gate pass 기준

질문:

- 실제 profitable tx를 gate가 잘못 막았는가
- 아니면 artifact만 잘못 남았는가

이 단계 전에는 `all-buy bootstrap`이나 큰 threshold 완화는 금지한다.

## P3. Freshness Follow-up

문제 1은 아직 닫히지 않았다.  
다음 지표를 계속 본다.

- `idleSkip delta`
- `idle_evicted`
- `candidate_seen -> candidate_evicted`
- `unique signaled tickers`
- top idle offenders

필요 시:

- idle diagnostics
- top idle offender 추적
- admission skip 분해

를 계속 보강한다.
