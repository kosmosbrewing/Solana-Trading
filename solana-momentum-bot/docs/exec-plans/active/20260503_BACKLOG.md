Latest implementation state:

- smart-v3 live entry no longer uses 24h `independentKolCount` as the live eligibility truth.
- Entry now computes a fresh same-mint KOL context from recent/candidate buys:
  - fresh independent KOL count
  - fresh S/A KOL count
  - fresh burst score
  - last buy age
  - pre-entry sell wave and fresh buys after the last sell
- Velocity live entry is the main live path: fresh 2+ KOLs, fresh S/A strength, fresh score threshold, and recent last buy are required.
- A+A fresh consensus is allowed for live velocity (`KOL_HUNTER_SMART_V3_VELOCITY_SCORE_THRESHOLD=5.0`), while S+B / A+B still fail the fresh S/A strength rule.
- Shadow/inactive KOLs are excluded from live fresh count/score. They are auxiliary only:
  - active 2+ live candidates can carry `SMART_V3_SHADOW_*` confirmation flags.
  - shadow-only candidates can still be measured as paper/shadow observations.
  - active 1 + shadow 1 cannot create live eligibility.
- Pullback remains measurable, but pure pullback live entry is a strategy reject by default via `SMART_V3_PULLBACK_LIVE_DISABLED` and no longer creates a main smart-v3 paper fallback row while live canary is active.
- Weak post-sell recovery is a strategy reject via `SMART_V3_POST_SELL_RECOVERY_WEAK` when prior sell flow is not followed by enough fresh independent re-buy.
- Live strict quality is fail-closed by default: `EXIT_LIQUIDITY_UNKNOWN`, `TOKEN_QUALITY_UNKNOWN`, `UNCLEAN_TOKEN*`, holder-risk, no-route, and rug-like flags block live unless an explicitly selected smart-v3 live-canary arm routes that exact bucket to live.
- 2026-05-13 paper/live divergence sprint added `smart_v3_fast_fail_live_v1` as an explicit, progressive live canary arm:
  - selected only through `KOL_HUNTER_LIVE_CANARY_ARMS`;
  - uses the same KOL live ticket/cap/floor as other canary arms;
  - routes paper-like smart-v3 strategy fallbacks to live for measurement, including live-disabled/pullback-disabled/weak post-sell/pre-entry sell-window/moderate quality fallback and bounded KOL-fill adverse cases;
  - still blocks no-route/rug-like flags, explicit high concentration, `HOLDER_TOP10_HIGH`, top10 concentration above its max threshold and survival max, repeated losing combos, and excessive KOL-fill adverse.
- 2026-05-13 pure_ws x smart-v3 paper join added `smart_v3_new_pool_confirmed_v1`:
  - paper-only shadow arm, never live by itself;
  - requires the same smart-v3 trigger plus recent new-pool context in `HeliusPoolRegistry` (`gecko_new_pool` / equivalent);
  - uses smart-v3 long-hold/tail-retain exit shape, not pure_ws fast scalp exits;
  - purpose is to test whether pure_ws/new-pool early discovery improves smart-v3 entry timing and MFE capture.
- Any pre-entry same-mint KOL sell now requires enough fresh independent re-buy plus a no-sell window before live; otherwise it is a strategy reject via `SMART_V3_PRE_ENTRY_SELL_LIVE_DISABLED` or `SMART_V3_RECENT_SELL_NO_SELL_WINDOW`.

  - Combo identity is fixed from entry-time fresh KOLs and stored as `smartV3EntryComboKey`.
  - Primary paper and live closes both feed the decay memory; live losses can fallback faster than paper-only samples.
  - Shadow arms are excluded from the decay memory.
- If fresh KOL fill-price data exists and our quote is materially above the KOL weighted entry, live is rejected via `SMART_V3_ENTRY_ADVANTAGE_ADVERSE`.
- Dev wallet status is now a live gate only:

  - `DEV_WALLET_ALLOWLIST` is telemetry only and never bypasses security, sell-route, drift, halt, or canary guards.
  - Missing/unknown dev status remains fail-open.

  - gates: 50/100 close sample thresholds, close-anchor based 30/60/300/1800 buy+sell T+ ok coverage, wallet/token-only/copyable PnL, and post-cost continuation
